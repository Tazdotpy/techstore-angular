1. Arquitectura del Proyecto

TechStore RD es una Single Page Application (SPA) construida con Angular 22. Toda la navegación ocurre en el cliente sin recargar la página; Angular intercepta los cambios de URL y muestra el componente correspondiente dentro del <router-outlet> del componente raíz.

techstore-angular/src/app/ ├── components/ ← Componentes reutilizables (UI global) │ ├── navbar/ (barra de navegación + carrito + tema) │ ├── footer/ (pie de página) │ ├── product-card/ (tarjeta de producto con @Input/@Output) │ ├── search-bar/ (búsqueda en tiempo real con debounce) │ └── category-menu/ (menú lateral de categorías) │ ├── pages/ ← Vistas completas (una por ruta) │ ├── home/ (página principal con productos destacados) │ ├── productos/ (catálogo con filtros reactivos) │ ├── detalle-producto/ (ficha de producto, ngSwitch) │ ├── carrito/ (tabla de items, cálculo ITBIS) │ ├── checkout/ (formulario reactivo + generación de orden) │ ├── contacto/ (formulario reactivo de contacto) │ ├── favoritos/ (★ Bonus — lista de favoritos) │ ├── historial/ (★ Bonus — historial de compras) │ ├── login/ (autenticación simulada) │ └── not-found/ (página 404) │ ├── services/ ← Lógica de negocio inyectable │ ├── producto.ts (HttpClient + fetch JSON) │ ├── carrito.ts (BehaviorSubject + LocalStorage) │ ├── auth.ts (sesión simulada) │ ├── tema.ts (dark mode toggle) │ ├── favoritos.ts (★ Bonus) │ └── historial.ts (★ Bonus) │ ├── models/ ← Interfaces TypeScript │ ├── producto.ts │ ├── carrito-item.ts │ └── orden.ts │ ├── guards/ ← Protección de rutas ├── interceptors/ ← Middleware HTTP ├── pipes/ ← Transformadores de datos │ ├── app.routes.ts ← Definición de rutas ├── app.config.ts ← Providers globales (Router, HttpClient) └── app.ts ← Componente raíz

Flujo de datos

Cuando el usuario agrega un producto al carrito: el ProductCard llama a CarritoService.agregar() → el servicio actualiza el BehaviorSubject → el Navbar recibe el nuevo conteo automáticamente gracias al AsyncPipe → los datos se persisten en localStorage.

2. Componentes y Templates

Un componente es la unidad básica de Angular. Combina lógica TypeScript, una plantilla HTML y estilos CSS en un bloque reutilizable e independiente.

Cada componente tiene tres partes: la clase TypeScript (lógica), el template HTML (vista) y el archivo SCSS (estilos encapsulados). El decorador @Component los conecta.
Usamos componentes para evitar repetir código. La tarjeta de producto (ProductCard) se reutiliza en Home, Productos y Favoritos sin duplicar HTML. 
// src/app/components/product-card/product-card.ts
@Component({
  selector: 'app-product-card',      // etiqueta HTML personalizada
  imports: [RouterLink, NgClass, NgIf, NgFor, DecimalPipe, UpperCasePipe],
  templateUrl: './product-card.html', // HTML separado
  styleUrl: './product-card.scss'     // CSS encapsulado (no contamina otros)
})
export class ProductCard implements OnInit {
  @Input() producto!: Producto;       // recibe datos del componente padre
  @Output() agregado = new EventEmitter<Producto>();  // notifica al padre
}
TechStore tiene 5 componentes compartidos (navbar, footer, product-card, search-bar, category-menu) y 10 páginas como componentes de ruta. Angular 22 usa standalone components — cada componente declara sus propias dependencias en el array imports.

3. Data Binding — Los 4 Tipos

El data binding es el mecanismo que sincroniza la clase TypeScript con el template HTML. Angular soporta 4 tipos:

3.1 Interpolación {{ }}

Muestra el valor de una variable del componente en el HTML. Angular evalúa la expresión y la convierte en texto. 
<!-- En product-card.html -->
<h3 class="card-nombre">{{ producto.nombre }}</h3>
<span>RD$ {{ producto.precio | number:'1.0-0' }}</span>

3.2 Property Binding [ ]

Vincula una propiedad del DOM con una expresión TypeScript. Los corchetes indican que el valor fluye del componente al HTML. 
<!-- En product-card.html -->
<img [src]="producto.imagen" [alt]="producto.nombre">

<!-- En navbar.html — activa/desactiva el menú hamburguesa -->
<div class="navbar-collapse" [class.show]="menuAbierto">

3.3 Event Binding ( )

Escucha eventos del DOM (clicks, inputs, etc.) y ejecuta un método del componente. Los paréntesis indican que el valor fluye del HTML al componente. 
<!-- En product-card.html -->
<button (click)="agregarAlCarrito()">🛒 Agregar</button>
<button (click)="toggleFavorito()">❤️</button>

<!-- En carrito.html — escucha cambios en el input -->
<input type="number" (change)="cambiarCantidad(item.producto.id, $event)">

3.4 Two-way Binding [( )]

Combina property binding y event binding. El valor fluye en ambas direcciones: si cambia el input del usuario, cambia la variable; si cambia la variable en el código, se actualiza el input. Requiere FormsModule. 
<!-- En search-bar.html — la búsqueda actualiza 'termino' en tiempo real -->
<input
  type="search"
  [(ngModel)]="termino"
  (input)="onInput()"
  placeholder="Buscar productos...">
// search-bar.ts — 'termino' se actualiza automáticamente con cada tecla
export class SearchBar {
  termino = '';  // sincronizado con el input
}

4. Directivas

Las directivas son instrucciones en el HTML que le dicen a Angular cómo transformar el DOM.

4.1 Directivas Estructurales — Modifican el DOM

*ngIf — Mostrar u ocultar elementos condicionalmente

Si la condición es true, Angular agrega el elemento al DOM. Si es false, lo elimina completamente (no solo lo oculta con CSS). 
<!-- En navbar.html — muestra badge solo si hay items -->
<span class="carrito-badge" *ngIf="(cantidadCarrito$ | async) as n">
  {{ n }}
</span>

<!-- En product-card.html — muestra precio original solo si existe -->
<span class="precio-original" *ngIf="producto.precioOriginal">
  RD$ {{ producto.precioOriginal | number:'1.0-0' }}
</span>

*ngFor — Repetir elementos sobre una lista

Itera sobre un array y genera un elemento HTML por cada ítem. El trackBy mejora el rendimiento indicando a Angular cómo identificar cada ítem para no re-renderizar todo. 
<!-- En productos.html — genera una card por producto -->
<app-product-card
  *ngFor="let p of lista; trackBy: trackById"
  [producto]="p">
</app-product-card>

<!-- En carrito.html — genera una fila por item del carrito -->
<tr *ngFor="let item of items; trackBy: trackById">
  <td>{{ item.producto.nombre }}</td>
  <td>{{ item.cantidad }}</td>
</tr>

*ngSwitch — Selección múltiple de plantillas

Evalúa una expresión y muestra el bloque que coincide con el valor. Equivalente al switch/case de JavaScript pero en el template. 
<!-- En detalle-producto.html — muestra estado según el stock -->
<div [ngSwitch]="producto.stock">
  <span *ngSwitchCase="0" class="stock-cero">⚠ Sin stock</span>
  <span *ngSwitchCase="1" class="stock-bajo">⚡ Último en stock</span>
  <span *ngSwitchDefault class="stock-ok">✓ {{ producto.stock }} disponibles</span>
</div>

4.2 Directivas de Atributo — Modifican apariencia o comportamiento

[ngClass] — Clases CSS dinámicas

<!-- En product-card.html — aplica clase si no hay stock -->
<div class="product-card" [ngClass]="{'sin-stock': producto.stock === 0}">

<!-- En category-menu.html — marca la categoría activa -->
<li *ngFor="let cat of categorias"
    [ngClass]="{'activa': cat.id === categoriaActual}">

5. Servicios e Inyección de Dependencias

Un servicio es una clase que encapsula lógica de negocio reutilizable (cargar datos, gestionar el carrito, autenticación, etc.) separada de los componentes. La inyección de dependencias (DI) es el mecanismo por el cual Angular provee automáticamente las instancias de servicios a quien las necesita.

Si la lógica del carrito estuviera en los componentes, habría que duplicarla en ProductCard, CarritoPage, Navbar, etc. Con un servicio singleton (providedIn: 'root'), todos comparten la misma instancia y el mismo estado. 
// src/app/services/carrito.ts
@Injectable({ providedIn: 'root' })  // singleton disponible en toda la app
export class CarritoService {
  private _items = new BehaviorSubject<ItemCarrito[]>([]);

  // Observable que los componentes pueden subscribir
  items$ = this._items.asObservable();

  agregar(producto: Producto): void {
    const items = [...this._items.value];
    // ... lógica de negocio
    this._items.next(items);  // emite el nuevo estado a todos los observadores
  }
}
// Cualquier componente lo inyecta con inject()
export class ProductCard {
  private carritoService = inject(CarritoService);  // Angular provee la instancia

  agregarAlCarrito(): void {
    this.carritoService.agregar(this.producto);
  }
}
TechStore tiene 6 servicios: ProductoService (HTTP + RxJS), CarritoService (BehaviorSubject + localStorage), AuthService (sesión simulada), TemaService (dark mode), FavoritosService y HistorialService (ambos bonus).

6. Routing — Navegación SPA

El Router de Angular permite navegar entre "páginas" sin recargar el navegador. Intercepta cambios de URL y renderiza el componente correspondiente dentro del <router-outlet>.

El archivo app.routes.ts define el mapa de rutas. Cuando la URL cambia a /carrito, Angular destruye el componente anterior y crea el CarritoComponent en el outlet. 
// src/app/app.routes.ts
export const routes: Routes = [
  // Carga inmediata (eager) — Home siempre disponible al arrancar
  { path: '', component: Home },

  // Lazy loading — solo se descarga cuando el usuario navega a /productos
  {
    path: 'productos',
    loadComponent: () =>
      import('./pages/productos/productos').then(m => m.Productos)
  },

  // Ruta dinámica — :id es un parámetro que cambia por cada producto
  { path: 'producto/:id',
    loadComponent: () => import('./pages/detalle-producto/detalle-producto')
      .then(m => m.DetalleProducto)
  },

  // Ruta protegida — el Guard verifica autenticación antes de entrar
  {
    path: 'checkout',
    loadComponent: () => import('./pages/checkout/checkout').then(m => m.Checkout),
    canActivate: [authGuard]
  },

  // Wildcard — captura cualquier URL no reconocida
  { path: '**', component: NotFound }
];

RouterLink y RouterLinkActive

<!-- En navbar.html -->
<a routerLink="/productos" routerLinkActive="activo">Catálogo</a>
<!-- routerLinkActive agrega la clase 'activo' automáticamente cuando la ruta coincide -->

<!-- Navegación programática desde TypeScript -->
constructor(private router: Router) {}
irAProducto(id: string): void {
  this.router.navigate(['/producto', id]);
}

Leer parámetros de la URL

// En detalle-producto.ts — lee el :id de la URL
export class DetalleProducto implements OnInit {
  private route = inject(ActivatedRoute);

  ngOnInit(): void {
    this.producto$ = this.route.params.pipe(
      switchMap(params => this.productoService.obtenerProducto(params['id']))
    );
  }
}

7. Lazy Loading

El Lazy Loading divide la aplicación en módulos que se descargan bajo demanda, no todos de golpe al arrancar. Esto reduce el tiempo de carga inicial.

Sin lazy loading, el navegador descargaría el código de todas las páginas aunque el usuario solo visite el home. Con lazy loading, checkout.js solo se descarga cuando el usuario navega a /checkout.
TechStore tiene 8 chunks lazy generados automáticamente: productos (25.9 kB), detalle-producto (24.4 kB), checkout (22.3 kB), carrito (19.6 kB), contacto (16.2 kB), historial (11.4 kB), login (9.6 kB), favoritos (6.2 kB). El bundle inicial es solo ~42 kB. 
// Sintaxis de lazy loading con loadComponent (Angular 15+)
{
  path: 'productos',
  loadComponent: () =>
    import('./pages/productos/productos').then(m => m.Productos)
  // ↑ El import() es una función asíncrona — el archivo solo se descarga
  //   cuando el usuario navega a /productos por primera vez
}

8. Comunicación entre Componentes (@Input / @Output)

Los componentes se comunican con sus padres e hijos mediante @Input y @Output.

@Input — Del padre al hijo

El componente padre pasa datos al hijo mediante property binding. El hijo declara qué datos acepta con @Input(). 
// product-card.ts (hijo)
@Input() producto!: Producto;  // recibe el producto del padre

// productos.html (padre) — pasa cada producto a la card
<app-product-card [producto]="p"></app-product-card>

@Output + EventEmitter — Del hijo al padre

El hijo emite un evento con EventEmitter. El padre lo escucha con event binding y ejecuta su propia lógica. 
// product-card.ts (hijo) — emite cuando el usuario agrega al carrito
@Output() agregado = new EventEmitter<Producto>();
@Output() favoritoToggled = new EventEmitter<Producto>();

agregarAlCarrito(): void {
  this.carritoService.agregar(this.producto);
  this.agregado.emit(this.producto);  // notifica al padre
}

// search-bar.ts (hijo) — emite el término de búsqueda
@Output() busqueda = new EventEmitter<string>();

// productos.html (padre) — escucha el evento del hijo
<app-search-bar (busqueda)="onBusqueda($event)"></app-search-bar>
<app-category-menu (categoriaSeleccionada)="onCategoria($event)"></app-category-menu>

9. Formularios Reactivos

Los Formularios Reactivos definen la estructura y validaciones del formulario en TypeScript (no en el HTML), haciendo que el estado del formulario sea predecible y testeable.

Los formularios reactivos permiten validar en tiempo real, desactivar el botón de envío hasta que el formulario sea válido, y mostrar mensajes de error dinámicos — sin lógica en el HTML. 
// checkout.ts — construcción del formulario en TypeScript
this.form = this.fb.group({
  nombre:    ['', [Validators.required, Validators.minLength(3)]],
  email:     ['', [Validators.required, Validators.email]],
  direccion: ['', [Validators.required, Validators.minLength(10)]],
  telefono:  ['', [Validators.required, Validators.pattern(/^\d{10,15}$/)]],
  tarjeta:   ['', [Validators.required, Validators.pattern(/^\d{16}$/)]]
});

// Acceder a los controles
get f() { return this.form.controls; }

// Enviar solo si el formulario es válido
onSubmit(): void {
  if (this.form.invalid) {
    this.form.markAllAsTouched();  // muestra todos los errores
    return;
  }
  // procesar orden...
}
<!-- checkout.html — el HTML solo refleja el estado del formulario -->
<form [formGroup]="form" (ngSubmit)="onSubmit()">
  <input formControlName="email" class="form-input"
         [class.error]="f['email'].invalid && f['email'].touched">

  <!-- Mensaje de error dinámico -->
  <span *ngIf="f['email'].invalid && f['email'].touched">
    Email válido requerido
  </span>

  <button type="submit">Confirmar Orden</button>
</form>

10. Observables y RxJS

Un Observable es un flujo de datos asíncrono que puede emitir múltiples valores a lo largo del tiempo. RxJS es la librería de operadores para transformar y combinar esos flujos.

Piensa en un Observable como una tubería de agua: los datos fluyen por ella. Los operadores (map, filter, tap) son válvulas que transforman el flujo. El subscribe() es la llave que abre la tubería para recibir el agua.

BehaviorSubject — Estado reactivo en tiempo real

Un BehaviorSubject es un Observable especial que: (1) recuerda el último valor emitido, (2) lo da inmediatamente a cualquier nuevo subscriptor, (3) puede actualizarse con .next(). Es ideal para representar estado que múltiples componentes necesitan observar. 
// carrito.ts — el carrito como estado reactivo
private _items = new BehaviorSubject<ItemCarrito[]>([]);

// Observable derivado — se recalcula automáticamente cuando _items cambia
total$ = this._items.pipe(
  map(items => items.reduce((sum, i) => sum + i.producto.precio * i.cantidad, 0))
);

cantidad$ = this._items.pipe(
  map(items => items.reduce((sum, i) => sum + i.cantidad, 0))
);

// Actualizar el estado (todos los observadores se notifican automáticamente)
agregar(producto: Producto): void {
  const nuevoEstado = [...this._items.value, { producto, cantidad: 1 }];
  this._items.next(nuevoEstado);
}

combineLatest — Combinar múltiples flujos

// productos.ts — filtra reactivamente cuando cambia búsqueda O categoría
this.productos$ = combineLatest([
  this.productoService.obtenerProductos(),  // Observable del HTTP
  this._busqueda$,    // BehaviorSubject del input de búsqueda
  this._categoria$    // BehaviorSubject del menú de categorías
]).pipe(
  map(([lista, busqueda, categoria]) => {
    // Se ejecuta cada vez que CUALQUIERA de los tres cambia
    let resultado = lista;
    if (categoria !== 'todos') resultado = resultado.filter(p => p.categoria === categoria);
    if (busqueda)              resultado = resultado.filter(p => p.nombre.includes(busqueda));
    return resultado;
  }),
  tap(lista => this.totalResultados = lista.length)  // efecto secundario sin modificar el flujo
);

switchMap — Aplanar Observables anidados

// detalle-producto.ts — cuando cambia el :id en la URL, hace nuevo request HTTP
this.producto$ = this.route.params.pipe(
  switchMap(params => this.productoService.obtenerProducto(params['id']))
  // switchMap cancela la petición anterior si el usuario navega rápido
);

AsyncPipe — Suscripción automática en el template

El AsyncPipe se suscribe al Observable en el HTML y se desuscribe automáticamente cuando el componente se destruye, evitando memory leaks. 
<!-- navbar.html — muestra el conteo del carrito, se actualiza solo -->
<span *ngIf="(cantidadCarrito$ | async) as n">{{ n }}</span>

<!-- productos.html — muestra los productos cuando llegan del servidor -->
<div *ngIf="productos$ | async as lista; else cargando">
  <app-product-card *ngFor="let p of lista" [producto]="p"></app-product-card>
</div>
<ng-template #cargando>Cargando...</ng-template>

11. HttpClient — Consumo de API REST

Angular provee HttpClient para hacer peticiones HTTP de forma reactiva (devuelve Observables). TechStore lo usa para cargar los productos desde /productos.json.

// src/app/app.config.ts — activar HttpClient globalmente
export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideHttpClient(withInterceptors([authInterceptor]))
                                   // ↑ conecta el interceptor automáticamente
  ]
};
// src/app/services/producto.ts
@Injectable({ providedIn: 'root' })
export class ProductoService {
  private http = inject(HttpClient);

  obtenerProductos(): Observable<Producto[]> {
    return this.http.get<Producto[]>('/productos.json').pipe(
      tap(lista => console.log(`${lista.length} productos cargados`)),
      catchError(err => {
        console.error('Error:', err);
        return of([]);  // devuelve array vacío en caso de error
      })
    );
  }

  obtenerProducto(id: string): Observable<Producto | undefined> {
    return this.obtenerProductos().pipe(
      map(lista => lista.find(p => p.id === id))
    );
  }
}

// src/app/pipes/descuento-pipe.ts
@Pipe({ name: 'descuento' })
export class DescuentoPipe implements PipeTransform {
  transform(precio: number, porcentaje: number = 0): number {
    if (!precio || porcentaje <= 0) return precio;
    return Math.round(precio * (1 - porcentaje / 100));
  }
}

private persistir(items: ItemCarrito[]): void {

private cargarDesdeStorage(): ItemCarrito[] {

14. Guards e Interceptors

Guards — Protección de Rutas

Un Guard es una función que Angular ejecuta antes de activar una ruta. Si devuelve false, cancela la navegación. Si devuelve un UrlTree, redirige a otra ruta.

En TechStore, el authGuard protege /checkout e /historial. Si el usuario no ha iniciado sesión, lo redirige a /login automáticamente. 
// src/app/guards/auth-guard.ts
export const authGuard: CanActivateFn = () => {
  const authService = inject(AuthService);
  const router = inject(Router);

  if (authService.estaAutenticado()) {
    return true;  // deja pasar
  }

  router.navigate(['/login']);  // redirige al login
  return false;
};

// En app.routes.ts — aplicar el guard a la ruta
{ path: 'checkout', ..., canActivate: [authGuard] }

Interceptors — Middleware HTTP

Un Interceptor intercepta todas las peticiones HTTP salientes y/o respuestas entrantes. Se usa para agregar headers de autenticación, manejar errores globalmente, o mostrar un loader.

// src/app/interceptors/auth-interceptor.ts
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = localStorage.getItem('techstore_token');

  // Clonar la petición y agregar el header Authorization si hay token
  const reqModificado = token
    ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })
    : req;

  return next(reqModificado).pipe(
    catchError(error => {
      // Manejo global de errores HTTP
      console.error(`Error HTTP ${error.status}:`, error.message);
      return throwError(() => error);
    })
  );
};

★ Tema Oscuro (Dark Mode)

El TemaService usa un BehaviorSubject para rastrear el tema actual. Al hacer toggle, agrega/quita la clase .tema-claro del elemento <html>. Las variables CSS en :root definen el tema oscuro (default) y .tema-claro las sobreescribe.

// tema.ts
toggleTema(): void {
  const nuevo = !this._oscuro.value;
  this._oscuro.next(nuevo);
  localStorage.setItem('techstore_tema', nuevo ? 'oscuro' : 'claro');
  document.documentElement.classList.toggle('tema-claro', !nuevo);
}

★ Lista de Favoritos

El FavoritosService mantiene un BehaviorSubject<Producto[]>. El botón de corazón en cada ProductCard llama a toggleFavorito() que agrega o quita el producto del array y lo persiste en localStorage. La página /favoritos muestra el Observable directamente con el AsyncPipe.

★ Historial de Compras

Al confirmar una orden en checkout, se construye un objeto Orden con ID único (ORD-timestamp-random), fecha, cliente, items, subtotal, ITBIS y total. Se pasa a HistorialService.agregarOrden() que lo prepende al array del BehaviorSubject y lo persiste en localStorage. La página /historial lista las órdenes con DatePipe para formatear las fechas.
